# Suffix vs Prefix: Choosing a Docker Port Convention That Actually Works

![Two cartoon Docker whales fighting over a single yellow port panel with tangled cables, illustrating two containers trying to bind the same host port](/blog/docker-port-convention-hero.png)

> **TL;DR:** Appending a project digit to Docker host ports feels clean on paper. Then it overflows 65535 and Docker refuses to start. Prepend the digit instead. The math always works, the numbers stay readable, and you will not find out the hard way.

One of those afternoons when I was trying to bring up a second Docker project on my laptop and the terminal kept shouting at me about port conflicts. Classic.

If you have ever run more than one compose stack on the same machine, you already know the script. You bring up Project A. It happily grabs port 3306 for MySQL. You try to bring up Project B. It also wants 3306. Docker shrugs and walks away. You go stop Project A. You forget it was running something you actually needed. The cycle continues.

So I needed a simple rule. Give every project its own single digit. Stick that digit on the port somewhere. Now nothing fights.

The only question left was: stick it at the front, or stick it at the back?

## The suffix idea

My first instinct was to go with a suffix. If MySQL is `3306` and the project tag is `3`, the host port becomes `33063`. Postgres `5432` becomes `54323`. It reads nicely from left to right. You can still see the original port at the start, then the project tag at the end, like a little name badge.

I went ahead and applied it to one of my stacks. Saved the compose file. Ran it.

What happened was this.

```bash
$ docker compose up
Error: invalid hostPort: 80803
```

I sat there for a good bit of time wondering what was wrong with `80803`. Then it clicked. `80803` is not a port. It cannot be a port. No port on the planet can be `80803`.

And just like that, the convention I liked the most started falling apart.

## The 16-bit reality check

Here is the bit I forgot. TCP and UDP ports are 16-bit unsigned integers. The highest valid port is `65535`. This is not a Docker rule. It is not a Linux rule. It is baked into how the protocol was designed decades ago, and nothing is going to change it anytime soon.

Now apply that to the suffix idea. Under the hood you are computing:

```
newPort = (oldPort * 10) + projectDigit
```

The moment your original port crosses `6553`, that formula spits out a number above `65535`. The kernel cannot bind it. Docker will not even accept it in the compose file. Your stack is stuck before it gets a chance to start.

Here is how that plays out across common dev ports:

```
8080 (phpmyadmin, tomcat, jenkins)  ->  80803   invalid
8081 (common alt-http)              ->  80813   invalid
8082 (common alt-http)              ->  80823   invalid
8025 (mailhog ui)                   ->  80253   invalid
9000 (php-fpm, minio, portainer)    ->  90003   invalid
9200 (elasticsearch)                ->  92003   invalid
6379 (redis)                        ->  63793   invalid
```

That is a pile of everyday services that simply cannot be remapped with a suffix. The convention only survives for a handful of ports below `6554`. Not exactly the clean rule I was hoping for.

Tell me I am not the only one who has committed to something that looked cleaner on paper and then got punished by it the moment real data showed up.

## Why prefix quietly wins

Flip the convention around and the problem goes away. Prepend the project digit instead of appending it. For a 4-digit original port you are now computing:

```
newPort = (projectDigit * 10000) + oldPort
```

Every one of the ports above now has a valid prefixed version:

```
8080  ->  38080   valid
8081  ->  38081   valid
9000  ->  39000   valid
9200  ->  39200   valid
6379  ->  36379   valid
```

No overflow. No special cases hiding in the second half of the port range. No "wait, is this one safe to suffix" moments.

## More than just math

Even putting the overflow aside, prefix ends up being nicer to live with day to day.

The original port stays readable at the tail. When I see `38080`, my brain parses it as "something on 8080, project 3". When I see `80803` my brain has to work backwards to figure out where the original port ended. A small difference, but you look at these numbers a lot.

You also get free grouping. Every port on one stack starts with `3`. If another project picks `4`, every port on that stack starts with `4`. A quick `lsof -iTCP | grep :3` gives you everything for one project. `docker ps` lines up nicely when you sort by port. Tiny quality of life stuff, but tiny quality of life stuff is the reason conventions exist at all.

And there is no math to second-guess. Whichever port you are mapping, prefix is fine. You do not have to stop and check whether today's port happens to be above `6553`.

## What the final mapping looked like

After switching the convention, the stack came back up on the first try. Here is a sample mapping using project digit `3`.

| Service      | Container port | Host port (prefix 3) |
|--------------|----------------|----------------------|
| Web (HTTP)   | 80             | 380                  |
| SSH          | 22             | 322                  |
| MySQL        | 3306           | 33306                |
| phpMyAdmin   | 80             | 38080                |
| MailHog UI   | 8025           | 38025                |
| MailHog SMTP | 1025           | 31025                |

One thing worth calling out: I did not have to touch any Apache site config or `.env` file. Containers talk to each other on their original ports inside the Docker network. Only the host side of the mapping changed. So `web -> db:3306` keeps working as before, even though on the laptop you now hit `localhost:33306`.

## One honest caveat

Prefix is not a silver bullet. If your original port is above `35535`, even prefix overflows. MongoDB on `27017` becomes `327017` with prefix, which is also invalid. For those few cases you have to pick a fresh memorable port rather than applying the rule blindly. Rare enough in practice that you deal with it case by case, but worth naming so nobody gets blindsided.

## The takeaway

If you are setting up a port convention for running multiple Docker projects side by side, save yourself the debate. Go with prefix. The math always works, the numbers stay readable, and nobody is going to discover the 16-bit limit in the middle of something important.
